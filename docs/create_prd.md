# create_prd.md — 학교 급식 알레르기 관리 시스템 - 알라리알라 (Aller Re-Alert)

> **이 문서의 용도:**
> LLM(AI 코딩 에이전트)에게 프로젝트 착수 전 전달하는 컨텍스트 문서이다.
> LLM은 이 문서를 읽고 프로젝트의 목적·범위·사용자·기능·제약조건을 이해한 뒤,
> 개발 계획을 수립하고 일관된 기준으로 작업을 수행해야 한다.
>
> **포함 범위:** 프로젝트 배경, 문제-해결 매핑, 사용자 정의, 요구사항 전체, 유저 스토리, 화면 흐름, 핵심 비즈니스 로직, 데이터 모델, 개발 페이즈
> **미포함 (별도 문서로 제공 예정):** 기술 스택, API 설계, 코딩 컨벤션, 인프라 구성

## 1. 프로젝트 개요

### 1.1 한 줄 요약

학교급식(향후 사내 식당·복지관 등 단체 급식으로 확장 가능)에서 **알레르기 보유 이용자의 안전을 보장**하고, **영양사(식단 담당자)의 식단 작성 업무를 AI로 보조**하며, **대체식 수요를 설문·투표로 자동 집계**하는 웹 애플리케이션이다.

### 1.2 프로젝트명

**알라리알라 (Aller Re-Alert)**

---

## 2. 문제 & 해결 전략

### 2.1 현재 문제 (AS-IS)

| # | 문제 | 대상 | 위험도 |
|---|------|------|--------|
| P1 | 알레르기 보유 이용자가 매일 식단표를 수동 확인해야 하며, 인적 오류로 알레르기 식품을 섭취할 위험이 있다 | 학생·교직원 | 매우 높음 |
| P2 | 대체식이 제공되어도 이용자에게 메뉴 선택권이 없어 만족도가 낮다 | 학생·교직원 | 중간 |
| P3 | 영양사가 정규 식단 + 대체 식단을 이중으로 작성해야 하여 업무가 과중하다 | 영양사 | 높음 |
| P4 | 알레르기 보유 이용자 수·유형 파악이 수동이라 대체식 수량 산정이 부정확하다 | 영양사 | 높음 |

### 2.2 해결 전략 (TO-BE)

| 문제 | 해결 전략 | 핵심 기능 |
|------|-----------|-----------|
| P1 | 시스템이 식단 × 알레르기 정보를 자동 대조하여 사전 알림 발송 | 알레르기 알림 엔진 |
| P2 | 대체식 필요 시 설문·투표로 이용자가 메뉴 선택에 참여 | 설문·투표 모듈 |
| P3 | AI가 영양 균형·예산·계절을 반영한 정규+대체 식단 초안 자동 생성 | AI 식단 생성 |
| P4 | 알레르기 정보 + 설문 응답 기반으로 수요 자동 집계 | 수요 대시보드 |

## 5. 유저 스토리 & 인수 조건

### 5.1 학생/교직원

```
US-01: 알레르기 정보 등록

  SO THAT 시스템이 자동으로 위험 식단을 걸러 알림을 보내준다

  인수 조건:
  - 식품위생법 기준 19종 알레르기 유발물질 체크리스트 제공

  관련: FR-USR-003, FR-USR-005
```

```
  SO THAT 한눈에 위험한 메뉴를 식별할 수 있다

  인수 조건:
  - 일간/주간/월간 뷰 전환 가능
  - 알레르기 포함 메뉴에 경고 아이콘 + 하이라이트
```

## 7. 핵심 비즈니스 로직

이 프로젝트의 안전 관련 핵심 로직이다. LLM은 이 로직을 정확히 구현해야 한다.

### 7.1 알레르기 알림 엔진 — 일일 알림

```
[매일 스케줄러 — 급식 N시간 전 실행]

1. 당일 공개된 식단(MealPlan, status=published) 조회
2. 식단에 연결된 전체 메뉴(MealItem)의 알레르기 태그 수집
   → meal_allergens = {난류, 대두, 밀, ...}
3. 해당 학교의 확정된 알레르기 정보(UserAllergen, status=confirmed) 전체 조회
4. 각 이용자별 교집합 계산:
   matched = user_allergens ∩ meal_allergens
5. matched가 비어있지 않은 이용자에게:
   a. Notification 생성 (type: allergy_alert)
   b. 푸시/이메일/SMS 발송 (이용자 설정에 따라)
   c. 해당 이용자가 학생이면 → 보호자에게도 동일 알림
6. 발송 결과 로깅
```

### 7.2 알레르기 알림 엔진 — 식단 변경 시 실시간 알림

```
[영양사가 공개된 식단 수정 시 트리거]

1. 수정 전 메뉴들의 allergen 집합 저장 → before_set
2. 수정 반영
3. 수정 후 allergen 집합 계산 → after_set
4. 새로 추가된 allergen = after_set - before_set
5. 새로 추가된 allergen에 해당하는 이용자 조회
6. 해당 이용자에게 즉시 알림 발송 (type: menu_change)
7. 변경 이력 기록
```

### 7.3 설문·투표 자동 생성 플로우

```
[영양사가 대체 식단 후보를 확정하고 마감일을 설정해 설문을 등록했을 때]

1. 대체 식단 후보의 대상 알레르기 유형 확인
2. 해당 알레르기를 보유한 이용자 목록 조회
3. 설문 자동 생성:
   a. Step 1 설문: "대체식이 필요한가요?" (예/아니오)
   b. Step 2 투표: 원래 식단 + 확정된 대체 식단 후보들 중 선택
      (Step 1에서 "예" 응답자만 참여)
4. 영양사가 지정한 마감일 설문에 적용
5. 대상 이용자에게 설문 참여 알림 발송
6. 마감일 도래 시:
   a. 미응답자에게 리마인더 발송
   b. 마감 후 결과 자동 집계
   c. 영양사 대시보드에 결과 반영
```

### 7.4 AI 식단 생성 시 프롬프트 구조

#### 정규 식단 초안 생성

```
[System 역할]
알라리알라(Aller Re-Alert)에서 영양사를 보조하는 AI.
한국 학교급식 영양 기준에 맞는 식단 설계.

[조건 파라미터]
- 기간, 1식 예산, 대상 학교/학년
- 1일 구성: 밥 1 + 국 1 + 반찬 3~4 + 후식 0~1
- 영양 기준(칼로리 범위), 선호/제외 식재료
- 기존 식단과 최소 N일 이상 비중복
- NEIS 급식 API 데이터: 해당 학교·기간의 실제 급식 메뉴 이력
  (나이스 교육정보 개방 포털 API 활용, 지역·학교별 메뉴 선호 반영 및 다양성 확보)

[출력]
날짜별 메뉴 구성 + 식재료 목록 + 알레르기 유발물질 + 영양 정보 (JSON)
```

> **NEIS 식단 API 활용 지침**
> - 나이스 교육정보 개방 포털(`open.neis.go.kr`)의 `mealServiceDietInfo` API로
>   학교별·날짜별 실제 급식 메뉴 이력 조회
> - AI 프롬프트에 최근 2~3개월치 이력 데이터를 컨텍스트로 제공하여
>   중복 방지·지역 식문화 반영·계절 식재료 선택에 활용
> - NEIS 데이터는 참고 자료이며, 최종 식단은 반드시 영양사가 검토·승인

#### 대체 식단 제안

```
[System 역할]
알라리알라(Aller Re-Alert) 대체 식단 제안 AI.

[입력]
원본 메뉴 정보 + 제외할 알레르기 유발물질
+ NEIS 급식 API 데이터 (해당 학교의 과거 대체식 메뉴 이력 참고)

[조건]
- 원본과 유사한 영양 구성 유지
- 제외 알레르기 유발물질이 결과에 절대 포함되지 않도록 검증
- 후보 2~3개 제안, 각각 영양 정보 포함
- 급식 환경에서 실현 가능한 메뉴
```

## 8. 데이터 모델 (ERD 개요)

### 8.1 핵심 엔티티

> **확장성 설계 원칙**: 현재는 학교급식을 1차 대상으로 구현하되,
> 추후 사내 식당·복지관·군대 급식 등 **학교 이외의 단체**로 확장할 수 있도록
> `School` 대신 `Organization` 추상화 엔티티를 사용한다.
> 학교 전용 필드(grade_structure 등)는 nullable로 처리한다.

```
User
  - id, org_id(FK), role(enum: member/staff/guardian/nutritionist/admin)
  - name, email, phone, group_info(학교: 학년/반, 기타: 부서/소속 등), created_at

Organization (단체 — 학교·사내 식당·복지관 등)
  - id, name, address
  - org_type(enum: school/company/welfare/military/other)
  - grade_structure(nullable, 학교 전용)
  - meal_time(급식 시간, 단체별 설정)

Allergen (마스터)
  - id, name, code(식약처코드), icon_url

UserAllergen (N:M 중간 테이블)
  - id, user_id(FK), allergen_id(FK)
  - status(enum: pending/confirmed/rejected)
  - approved_by(FK→User), created_at

MealPlan (하루 식단)
  - id, org_id(FK), date
  - status(enum: draft/published), published_at, created_by(FK)

MealItem (개별 메뉴)
  - id, meal_plan_id(FK)
  - category(enum: rice/soup/side/dessert), name
  - calories, nutrients(JSON)

MealItemAllergen (메뉴 × 알레르기 태깅)
  - meal_item_id(FK), allergen_id(FK), is_auto_tagged(boolean)

AlternateMealPlan (대체 식단)
  - id, meal_plan_id(FK), target_allergen_id(FK)
  - status(enum: proposed/confirmed), confirmed_by(FK)

AlternateMealItem (대체 메뉴)
  - id, alt_plan_id(FK), replaces_item_id(FK→MealItem)
  - name, calories, nutrients(JSON)

Survey (설문/투표)
  - id, meal_plan_id(FK), type(enum: need_check/menu_vote)
  - options(JSON)              ← 원래 식단 + 대체 식단 후보 목록 포함
  - deadline(영양사가 설문 등록 시 직접 설정)
  - status(enum: open/closed), created_by(FK)

SurveyResponse (설문 응답)
  - id, survey_id(FK), user_id(FK)
  - response(JSON), voted_item_id(FK), created_at, updated_at

Notification (알림)
  - id, user_id(FK), type(enum: allergy_alert/survey/approval/menu_change)
  - title, body, is_read, sent_at
```

### 8.2 핵심 관계

- Organization → User : 1:N (org_id FK)
- Organization → MealPlan : 1:N (org_id FK)
- User ↔ Allergen : N:M (UserAllergen, 보호자 승인 상태 포함)
- MealPlan → MealItem : 1:N
- MealItem ↔ Allergen : N:M (MealItemAllergen)
- MealPlan → AlternateMealPlan : 1:N
- AlternateMealPlan → AlternateMealItem : 1:N
- MealPlan → Survey : 1:N
- Survey → SurveyResponse : 1:N
- User → Notification : 1:N

## 11. 제약 조건 & 주의 사항

LLM이 계획 수립 및 구현 시 반드시 인지해야 할 사항이다.

### 11.1 안전 (최우선)

- **알레르기 알림 엔진은 이 시스템의 존재 이유**이다. 알림 누락은 이용자 건강에 직접적 위험을 초래한다. 이 모듈의 정확성·안정성이 가장 높은 우선순위를 가진다.
- 알레르기 정보와 식단 데이터의 **정합성이 깨지면 안 된다**. 트랜잭션 처리에 주의할 것.
- 시스템 장애 시에도 **수동 전환 수단**(PDF 다운로드 등)이 반드시 동작해야 한다.

### 11.2 개인정보

- 알레르기 정보는 **건강 관련 민감정보**이다. 별도 동의 필요.
- **14세 미만** 학생: 법정대리인(보호자) 동의 필수.
- 저장 시 암호화, 전송 시 TLS, 접근 시 RBAC.
- **로그에 알레르기 정보가 평문으로 노출되지 않도록** 할 것.

### 11.3 AI

- AI 생성 식단은 반드시 **영양사 검토·승인 후** 공개. 자동 공개 금지.
- AI 대체 식단 제안 시 **제외 대상 알레르기 식재료가 결과에 절대 포함되지 않도록** 검증 로직 필수.
- AI 응답의 **영양 기준 충족 여부를 자동 검증**하는 로직 포함할 것.

### 11.4 데이터 정합성

- 메뉴 추가/수정 시 알레르기 태깅 **자동 실행** + 영양사 수동 보정 가능.
- 설문·투표 마감 후 응답 변경 **불가**.
- 보호자 미승인(status≠confirmed) 알레르기 정보는 알림 엔진 대조 대상에서 **제외**.
